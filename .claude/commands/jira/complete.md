# Jira 티켓 완료

**사용법**: `jira-complete LAD-42`
- PR이 없으면 자동 생성, 있으면 업데이트

최종 검증 후 PR을 생성하고 티켓을 완료 처리합니다.

## 인자
- 첫 번째: Jira 티켓 ID (필수, 예: LAD-42)
- 두 번째: PR 번호 (선택, 이미 PR이 있는 경우만)
  - 보통은 생략 (자동으로 PR 생성)

## 실행 내용

$ARGUMENTS

### 단계 7.5: 테스트 실행 (PR 생성 전 필수)

PR 생성 전에 **반드시** 로컬 테스트를 실행한다.

#### 백엔드 테스트
```bash
cd backend && ./gradlew test --no-daemon
```
- ✅ 모든 테스트 통과 → 단계 8로 진행
- ❌ 테스트 실패 → **즉시 중단**, 실패 원인 분석 후 수정. 테스트가 통과될 때까지 PR 생성하지 않음
- ⚠️ DB 연결 오류 등 환경 문제 → 사용자에게 보고 후 진행 여부 확인

#### 프론트엔드 테스트 (변경된 경우)
```bash
cd frontend && npm run lint && npm run build
```
- ✅ lint + build 통과 → 단계 8로 진행
- ❌ 실패 → 즉시 수정

#### 테스트 결과 요약 출력
```
🧪 테스트 결과:
  - 백엔드: X개 통과 / Y개 실패 (실패 시 원인 목록)
  - 프론트엔드: lint ✅/❌, build ✅/❌
```

### 단계 8: 최종 검증
1. 모든 테스트 통과 확인 (단계 7.5에서 이미 실행)
2. 코드 품질 기준 충족 확인
3. 수락 조건(AC) 체크리스트 검토
4. 변경사항 최종 요약

### 단계 8.5: docs/ 문서 관리
작업 중 발생한 컨텍스트와 의사결정을 `docs/` 폴더에 반영합니다.

**1. 기존 문서 영향 분석**
- `docs/` 하위 파일들을 스캔하여 이번 변경사항과 관련된 문서 식별
- 변경으로 인해 outdated된 내용이 있는지 확인

**2. 문서 업데이트 (해당 시)**
- 기존 문서 중 이번 작업으로 영향받는 내용 업데이트
  - 예: 아키텍처 변경 → 관련 설계 문서 수정
  - 예: 새 API 추가 → API 가이드 업데이트
  - 예: 인프라 변경 → 배포/운영 가이드 수정

**3. 새 문서 작성 (해당 시)**
- 아래 기준에 해당하면 새 문서를 `docs/`에 생성:
  - 새로운 아키텍처 패턴이나 기술 도입
  - 중요한 기술적 의사결정 (ADR 형태)
  - 향후 참조가 필요한 설정/구성 가이드
  - 복잡한 비즈니스 로직이나 도메인 규칙
- 문서 네이밍: kebab-case (예: `ai-course-generation-flow.md`)

**4. 문서 작성 기준**
- 헤더에 메타정보 포함: 프로젝트명, 작성일, 상태, 관련 티켓
- 의사결정 배경(Why)과 결론(What) 중심으로 작성
- 대안 검토 내용이 있었다면 함께 기록
- Mermaid 다이어그램 적극 활용 (아키텍처, 흐름도)

**5. 변경 없는 경우**
- 단순 버그 수정 등 문서 영향이 없으면 이 단계 스킵
- PR 본문에 "docs 변경 불필요" 표기

### 단계 9: PR 생성 및 티켓 완료
1. 원격 브랜치로 push
2. PR 생성 (gh cli 사용)
   - 제목: `[티켓ID] {티켓 제목}`
   - 본문: 변경사항 요약, 테스트 결과, AC 체크리스트
3. Jira 티켓 상태 업데이트 (Done / In Review)
4. PR 링크를 Jira 티켓에 연결

### PR 템플릿 내용
```markdown
## Summary
- Jira 티켓: [티켓ID](티켓 링크)
- 변경 요약

## Changes
- 주요 변경사항 목록

## Test Plan
- [ ] 단위 테스트 통과
- [ ] 통합 테스트 통과
- [ ] E2E 테스트 통과

## Acceptance Criteria
- [ ] AC 항목들...
```

### 출력
- PR URL
- 최종 변경사항 요약
- Jira 티켓 상태
- 리뷰어 할당 안내
