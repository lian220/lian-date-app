# Jira 티켓 생성

**사용법**:
- `jira:create "티켓 제목"` - 기존 티켓 패턴 자동 분석 후 생성
- `jira:create "티켓 제목" --ref LAD-42` - 특정 티켓을 템플릿으로 사용
- `jira:create "티켓 제목" --ref LAD-42,LAD-43` - 여러 티켓의 공통 요소 추출
- `jira:create --from LAD-42` - 기존 티켓을 그대로 복제 후 수정

기존 티켓들의 요소(이슈 타입, 레이블, 컴포넌트, 스프린트, 우선순위, AC 형식 등)를 분석하고
일관된 형식으로 새 티켓을 생성합니다.

## 인자

$ARGUMENTS

- **첫 번째**: 새 티켓 제목 (선택)
  - 생략 시 대화형으로 제목 입력 요청
- **--ref [ID,...]**: 참조할 기존 티켓 ID (쉼표 구분, 복수 가능)
  - 생략 시 최근 완료된 티켓 3개를 자동으로 분석
- **--from [ID]**: 특정 티켓을 기반으로 복제 (제목만 변경)
- **--type [type]**: 이슈 타입 강제 지정 (Story/Task/Bug/Sub-task)
- **--sprint**: 현재 활성 스프린트에 자동 할당

---

## 실행 단계

### 단계 1: 참조 티켓 분석

#### 1-1. 참조 티켓 결정
- `--ref` 인자가 있으면 해당 티켓들 조회
- `--from` 인자가 있으면 해당 티켓 1개만 조회 (복제 모드)
- 아무것도 없으면:
  1. `mcp__atlassian__searchJiraIssuesUsingJql`로 최근 완료 티켓 3-5개 조회
     - JQL: `project = LAD AND status = Done ORDER BY updated DESC`
  2. 조회된 티켓들을 참조 패턴으로 사용

#### 1-2. 각 참조 티켓에서 추출할 요소
`mcp__atlassian__getJiraIssue`로 각 티켓 조회 후 아래 요소 수집:

| 요소 | 필드명 | 설명 |
|------|--------|------|
| 이슈 타입 | `issuetype` | Story / Task / Bug / Sub-task |
| 레이블 | `labels` | 태그 목록 |
| 컴포넌트 | `components` | 시스템 컴포넌트 |
| 우선순위 | `priority` | Highest~Lowest |
| 스프린트 | `sprint` | 현재 스프린트 정보 |
| 스토리 포인트 | `story_points` / `customfield_*` | 예상 작업량 |
| 에픽 링크 | `epic` | 상위 에픽 |
| 담당자 | `assignee` | 현재 사용자 또는 기존 담당자 |
| AC 형식 | `description` | 수락 조건 작성 패턴 |
| 보고자 | `reporter` | 현재 사용자 |

#### 1-3. 공통 요소 추출 (복수 참조 시)
여러 티켓 분석 시 **빈도 기반 자동 추출**:
- 2개 이상 티켓에 공통으로 있는 레이블 → 신규 티켓에 자동 포함
- 모든 티켓에 동일한 컴포넌트 → 자동 포함
- 가장 자주 사용된 우선순위 → 기본값으로 제안
- 활성 스프린트 감지 → 자동 할당 여부 확인

---

### 단계 1.5: 대화형 추가 정보 수집

참조 티켓 분석이 끝나면, 자동으로 채울 수 없거나 확인이 필요한 항목을 **순서대로 질문**한다.
이미 인자로 받았거나 참조 티켓에서 명확히 추론된 항목은 건너뛴다.

#### 질문 흐름

**[Q1] 제목** — 인자로 받은 경우 생략
```
📝 티켓 제목을 입력해주세요:
```

**[Q2] 이슈 타입** — `--type` 플래그가 없고, 참조 티켓만으로 불명확한 경우
```
🏷️  이슈 타입을 선택해주세요:
  1) Story  - 사용자 관점의 기능 단위
  2) Task   - 기술적 작업 (리팩토링, 설정 등)
  3) Bug    - 버그 수정
  4) Sub-task - 상위 티켓의 하위 작업
→ (참조 티켓 기준 추천: [추천값]) 번호 또는 이름 입력:
```

**[Q3] 작업 배경 / 목적**
```
💬 이 티켓의 배경이나 목적을 간단히 설명해주세요.
   (설명(Description) 작성에 활용됩니다. 없으면 Enter 건너뜀)
→
```

**[Q4] 완료 조건 힌트 (AC)**
```
✅ 완료 조건(Acceptance Criteria)에 포함할 내용이 있으면 입력해주세요.
   (쉼표로 구분하거나 자유롭게 작성. 없으면 자동 생성)
→
```

**[Q5] 스토리 포인트** — 참조 티켓에 값이 없거나 범위가 넓은 경우
```
🔢 스토리 포인트를 입력해주세요 (1 / 2 / 3 / 5 / 8):
   (참조 티켓 평균: [평균값])
→
```

**[Q6] 에픽 연결** — 참조 티켓에서 에픽이 감지된 경우 확인, 없으면 질문
```
🗂️  연결할 에픽이 있으면 에픽 ID를 입력해주세요.
   (감지된 에픽: [에픽명] [에픽ID] / 없으면 새 에픽 자동 생성)
→
```
- 에픽 ID를 입력하면 해당 에픽에 연결
- **입력 없이 건너뛰면 새 에픽을 자동 생성**: 티켓 제목/내용 기반으로 에픽명 추론 후 `mcp__atlassian__createJiraIssue`로 에픽 먼저 생성, 이후 스토리에 `parent`로 연결

**[Q7] 기타 메모**
```
📎 기술 노트나 참고사항이 있으면 입력해주세요.
   (없으면 Enter 건너뜀)
→
```

#### 수집 완료 후
모든 입력값을 정리해서 단계 2로 넘김:
```
💾 입력 정보 수집 완료:
  - 제목: [값]
  - 이슈 타입: [값]
  - 배경: [값 or "자동 생성"]
  - AC 힌트: [값 or "자동 생성"]
  - 스토리 포인트: [값]
  - 에픽: [값 or "자동 생성"]
  - 기타 메모: [값 or "없음"]
```

### 단계 2: 새 티켓 내용 구성

#### 2-1. 제목 처리
- 인자로 받은 제목 사용
- 없으면 [Q1]에서 수집한 값 사용

#### 2-2. 설명 / AC 자동 생성
[Q3] 배경, [Q4] AC 힌트, [Q7] 기타 메모를 기반으로 작성.
참조 티켓의 수락 조건 형식을 그대로 따라 작성:

```
## 설명
[Q3 배경 입력값 또는 제목에서 자동 추론]

## 수락 조건 (Acceptance Criteria)
- [ ] AC 1: [Q4 힌트 기반 또는 자동 생성]
- [ ] AC 2: [Q4 힌트 기반 또는 자동 생성]
- [ ] AC 3: [Q4 힌트 기반 또는 자동 생성]

## 기술 노트
- [Q7 메모 입력값 또는 구현 고려사항 자동 생성]
```

#### 2-3. 스프린트 처리
- 참조 티켓에 활성 스프린트가 있으면 동일 스프린트에 할당
- `--sprint` 플래그가 없으면 할당 전 사용자에게 확인

#### 2-4. 생성 전 미리보기
사용자에게 아래 정보를 보여주고 확인 요청:

```
📋 생성할 티켓 미리보기:

제목: [입력된 제목]
이슈 타입: Story / Task / Bug
우선순위: Medium
레이블: [추출된 레이블들]
컴포넌트: [추출된 컴포넌트들]
스프린트: [스프린트명] (참조: LAD-42)
스토리 포인트: [추천값]
에픽: [에픽명]

수락 조건:
- [ ] AC 1: ...

[참조 티켓: LAD-42, LAD-43]

✅ 이대로 생성하시겠습니까? (수정이 필요한 항목을 말씀해주세요)
```

### 단계 3: Jira 티켓 생성

#### 3-1. 에픽 처리 (스토리 생성 전)
- [Q6]에서 기존 에픽 ID가 입력된 경우 → 해당 에픽 사용
- **입력 없는 경우 → 에픽 자동 생성**:
  1. 티켓 제목/내용 기반으로 에픽명을 **간결하고 명확하게** 추론
     - 좋은 예: `[Backend] Swagger API 문서화`, `[Frontend] 코스 결과 UI`, `[공통] 인증 시스템`
     - 나쁜 예: `## 개요\n\n백엔드 REST API의...` (설명 텍스트를 그대로 사용 금지)
     - 규칙: 10~30자 이내, prefix([Backend]/[Frontend]/[공통]) 포함, 기능 단위로 명명
  2. `mcp__atlassian__createJiraIssue`로 에픽 먼저 생성 (issueTypeName: "에픽")
  3. 생성된 에픽 ID를 스토리의 `parent`로 사용

#### 3-2. 티켓 생성
`mcp__atlassian__createJiraIssue`로 스토리 생성:
- 프로젝트: `LAD` (lian-ai-date)
- 위에서 구성한 모든 필드 포함
- 에픽 연결: 생성 후 `mcp__atlassian__editJiraIssue`로 `parent` 필드에 에픽 key 설정

#### 3-3. 추가 설정 (생성 후)
- 스프린트 할당 (지원되는 경우)
- 참조 티켓과의 연관 관계 설정 (선택)
  - "relates to" 링크로 참조 티켓과 연결

#### 3-3. 생성 결과 확인
`mcp__atlassian__getJiraIssue`로 생성된 티켓 검증

---

### 단계 4: 후속 작업 안내

생성 완료 후 바로 작업 시작 여부 확인:
- **Y**: `jira:start [새 티켓 ID]` 자동 실행
- **N**: 생성된 티켓 정보만 출력

```markdown
- LAD-42 (지역 선택 맵 구현) → 레이블, 컴포넌트 참조

🚀 다음 단계:
- 바로 작업 시작: `jira:start LAD-57`
- 나중에 시작: 티켓 ID를 메모해두세요
```

## 복제 모드 (--from)

`jira:create --from LAD-42 "새 기능명"` 실행 시:
1. LAD-42의 **모든 필드** 그대로 복사
2. 제목만 새 제목으로 교체
3. 상태는 "해야 할 일 (To Do)"으로 초기화
4. 담당자는 현재 사용자로 변경
5. 미리보기 후 생성 확인

## 에러 처리

- **티켓 조회 실패**: 수동으로 필드 입력 요청
- **스프린트 할당 실패**: 생성은 완료, 스프린트는 수동 할당 안내
- **필수 필드 누락**: 누락된 필드 목록과 함께 입력 요청
- **중복 제목 감지**: 유사한 기존 티켓 목록 표시 후 계속 여부 확인
