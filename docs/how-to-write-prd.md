# PRD (Product Requirements Document) 작성 가이드

## 📋 개요

이 문서는 모든 프로젝트에서 활용할 수 있는 PRD 작성 방법론과 가이드라인을 제공합니다.

> **활용 방법**: 이 가이드는 웹/앱 서비스, SaaS, 플랫폼, 모바일 앱 등 다양한 제품 유형에 적용 가능합니다. 각 섹션의 템플릿을 프로젝트 특성에 맞게 조정하여 사용하세요.

## 🎯 PRD의 목적

- **제품 비전 공유**: 팀 전체가 같은 목표를 향해 일하도록 정렬
- **요구사항 명확화**: 모호함 제거, 구현 범위 명확화
- **의사결정 기록**: 왜 이렇게 만들기로 했는지 문서화
- **개발 가이드**: 개발팀이 무엇을 어떻게 만들어야 하는지 안내

## 📁 PRD 구조

### 1. 배경 및 문제 정의
**목적**: 왜 이 기능/제품을 만드는가?

```markdown

### 1.1 왜 이 서비스를 만드는가?

**시장 기회**
- 타겟 시장의 현황과 기회 설명

**핵심 문제**
| 문제 | 현재 상황 | 영향 |
|------|----------|------|
| ... | ... | ... |

**우리의 솔루션**
- 문제를 어떻게 해결할 것인가
```

### 3. 제품 목표 및 성공 지표

> "한 문장으로 요약한 MVP 목표"

### 3.2 성공 지표 (HEART Framework)

| HEART | 지표명 | 정의 | MVP 목표 | 측정 방법 |

1. **지표 1** → 검증 목표

## 🎨 제품 유형별 적용 가이드

이 PRD 구조는 다양한 제품 유형에 맞게 조정할 수 있습니다:

### B2C 서비스 (소비자 대상)
- **페르소나**: 구체적인 개인 사용자 프로필 중심
- **성공 지표**: Engagement, Retention, Viral coefficient
- **특이사항**: 사용자 경험과 감성적 가치 강조

### B2B/SaaS
- **페르소나**: 조직 내 역할별 사용자 (의사결정자, 실무자, 관리자)
- **성공 지표**: ARR, Churn rate, NPS, Time-to-value
- **특이사항**: ROI, 통합성, 보안, 확장성 강조

### 플랫폼/마켓플레이스
- **페르소나**: 공급자와 수요자 양쪽 페르소나
- **성공 지표**: GMV, 거래 성사율, 양쪽 참여율
- **특이사항**: 네트워크 효과, 양면 시장 균형

### 모바일 앱
- **페르소나**: 사용 환경과 디바이스 특성 포함
- **성공 지표**: DAU, Session length, App store rating
- **특이사항**: 오프라인 기능, 푸시 알림, 배터리/데이터 사용

### 엔터프라이즈 솔루션
- **페르소나**: 조직 규모별, 산업별 특성
- **성공 지표**: 도입률, 사용자 교육 완료율, 지원 티켓 감소율
- **특이사항**: 온프레미스/클라우드, 규정 준수, 커스터마이징

## ✍️ PRD 작성 시 주의사항

### DO ✅
- **구체적으로**: "빠르게"가 아니라 "2초 이내"
- **측정 가능하게**: 성공을 어떻게 측정할지 명시
- **사용자 중심**: 기술이 아닌 사용자 가치 중심
- **Why 설명**: 왜 이렇게 만드는지 이유 명시
- **범위 명확히**: MVP 범위와 범위 외 구분

### DON'T ❌
- **모호한 표현**: "대략", "많은", "빠른" 등
- **How 상세 설명**: 구현 방법은 개발팀에 위임
- **모든 것 포함**: MVP 범위 초과 기능 포함
- **검증 없는 가정**: 가정은 반드시 근거 제시
- **일방적 작성**: 팀과 협업 없이 독단적 작성

## 🔗 연관 워크플로우

### PRD → Jira 티켓 생성

1. **PRD 작성 완료**
2. **기능별 Story 티켓 생성** (기능 1, 기능 2, 기능 3...)
3. **Story별 Sub-task 분해** (Frontend, Backend, Design 등)
4. **우선순위 및 스프린트 배정**

참고: `docs/jira-tickets-plan.md`

### Jira 티켓 → 개발 → 커밋

1. **티켓 시작** (Status: In Progress)
2. **개발 진행**
3. **커밋**: `.claude/commands/jira-commit.md` 참조
   ```bash
   jira-commit PROJECT-123 "feat(feature): 기능 구현"
   ```
4. **Jira 자동 업데이트** (작업 로그, 상태 전환)

## ❓ FAQ

### Q1: PRD는 누가 작성하나요?
**A**: 일반적으로 Product Manager가 작성하지만, 팀 협업으로 작성하는 것이 이상적입니다. 개발자, 디자이너, QA가 함께 리뷰하고 피드백을 주면서 완성합니다.

### Q2: PRD 작성에 얼마나 시간이 걸리나요?
**A**:
- **소규모 기능**: 2-4시간
- **중규모 프로젝트**: 1-2일
- **대규모 프로젝트**: 1주일 이상

중요한 것은 시간보다 품질입니다. 모호한 PRD는 개발 중 더 많은 시간을 낭비하게 만듭니다.

### Q3: PRD와 기술 명세서의 차이는?
**A**:
- **PRD (Product Requirements Document)**: **무엇을, 왜** 만들 것인가 (비즈니스 관점)
- **기술 명세서 (Technical Spec)**: **어떻게** 만들 것인가 (기술 관점)

PRD 작성 후 → 기술 명세서 작성 순서로 진행합니다.

### Q4: PRD는 언제 업데이트하나요?
**A**:
- 요구사항 변경 시
- 새로운 제약사항 발견 시
- MVP 범위 조정 시
- 주요 의사결정 변경 시

버전 관리를 통해 변경 이력을 추적합니다.

### Q5: AI로 PRD를 완전히 자동화할 수 있나요?
**A**: 아니요. AI는 초안 작성과 구조화에 도움을 주지만, 최종적으로 **사람의 판단과 검증**이 필수입니다. 특히:
- 비즈니스 전략과의 정렬
- 타겟 유저 검증
- 기술적 제약사항 고려
- 팀 역량 평가

이런 부분은 AI가 대체할 수 없습니다.
